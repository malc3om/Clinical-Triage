AntiGravity Research Lab — OpenEnv

Real-world domain Novel

An AI agent acts as a propulsion engineer running experiments on a simulated magnetic levitation & propulsion testbed. It must tune parameters, diagnose anomalies, and optimize flight profiles — tasks that mirror real aerospace R&D workflows.

Domain motivation

Magnetic levitation and propulsion research is a genuine, high-stakes engineering discipline. Real engineers spend hours iterating on coil configurations, power schedules, and stability controllers. No OpenEnv environment currently models this domain — strong novelty signal.

The environment simulates a MagLev Propulsion Test Rig: a linear track with controllable electromagnet arrays, a levitating payload, onboard sensors, and a fault injection system. The agent issues control commands and receives physics-grounded observations.

Action space

Actions are typed Pydantic models submitted via POST /step. Each action contains:

class Action(BaseModel): coil_power: List[float] # 0.0–1.0, per coil segment (8 values) stabilizer_gain: float # PID proportional gain 0.0–10.0 target_altitude: float # meters, 0.01–0.50 thrust_vector: Tuple[float,float] # (x, z) normalized direction diagnostic_probe: Optional[str] # sensor ID to query, or None
Observation space

Returned after each step — typed Pydantic model:

class Observation(BaseModel): altitude: float # current payload height (m) velocity: Tuple[float,float,float] # vx, vy, vz (m/s) coil_temperatures: List[float] # °C per coil magnetic_flux: List[float] # Tesla per segment fault_flags: List[str] # active system faults energy_consumed: float # joules this step stability_index: float # 0.0–1.0 time_step: int # episode step counter

Task 1 — Stable Levitation (Easy)
Score 0.0 – 1.0
Achieve and maintain stable hover at exactly 0.20 m altitude for 30 consecutive steps. The payload starts on the track (altitude = 0). The agent must ramp coil power, tune the stabilizer gain, and hold the target height without crashing or overheating any coil.
Deterministic grader
Partial credit
Grader formula: score = (steps_in_band / 30) × stability_bonus − overtemp_penalty
Success criteria: altitude within ±0.02 m of 0.20 m for all 30 steps, no coil temp > 85°C, no fault flags active. Score 1.0 if all met.
Task 2 — Fault Recovery (Medium)
Score 0.0 – 1.0
The environment injects a random single-coil fault at a hidden step (between steps 5–15). The agent must detect the fault via the diagnostic_probe action, identify which coil failed, redistribute power to compensate, and restore stable levitation within 20 steps of fault onset — all without losing the payload (altitude must never reach 0).
Deterministic grader
Multi-signal reward
Grader formula: score = detection_bonus + recovery_speed_score − altitude_loss_penalty
Success criteria: correct fault identified within 5 steps of onset (0.3 pts), altitude never drops below 0.05 m (0.3 pts), stable levitation restored (0.4 pts).
Task 3 — Minimum-Energy Transit (Hard)
Score 0.0 – 1.0
Navigate the payload from start position to a target position 2.0 m along the track in the minimum possible energy budget. The agent must plan a thrust schedule, manage altitude during acceleration/deceleration phases, avoid thermal runaway on any coil, and arrive within a 0.05 m position window — all within 60 steps. Optimal energy is pre-computed; score scales inversely with energy overspend.
Deterministic grader
Challenges frontier models
Grader formula: score = arrival_bonus × (E_optimal / E_actual) − crash_penalty
Success criteria: reaches target window (0.5 pts), energy efficiency ratio (0–0.4 pts), no coil failures during transit (0.1 pts). Score 0 if payload crashes.
Why these tasks genuinely challenge frontier models

Task 1 tests basic control loop understanding. Task 2 requires active probing — the agent must choose to spend a step on diagnostics rather than control, which is a non-obvious trade-off. Task 3 requires multi-step planning with a physics constraint (energy is path-dependent), which defeats naive greedy strategies.

Reward signal design

Rewards are dense throughout the trajectory — not just at episode end. Every step returns a float reward composed of weighted sub-signals.

Per-step reward components
Altitude closeness
+0.0 to +0.4
Stability index
+0.0 to +0.3
Energy efficiency
+0.0 to +0.2
Coil temp penalty
−0.05 per coil >80°C
Crash / fault penalty
−1.0 (episode ends)
Infinite loop penalty
−0.1 if no change in 5 steps
Why this satisfies the hackathon reward requirement

The reward is non-sparse: every step the agent gets a signal about how well it's doing on altitude, stability, and efficiency simultaneously. Partial progress is always rewarded — getting from 0 m to 0.15 m already scores positive reward even if the target is 0.20 m. Destructive behaviors (crashing, thermal runaway, infinite no-ops) are explicitly penalized. The hard task's energy-efficiency component means reward doesn't plateau — there's always room to improve.
